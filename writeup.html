Take-Home · Writeup

Extending Shotcut with CapCut-style features, driven by an agent over MCP

We added six CapCut-style editing capabilities to Shotcut (Qt / C++ / MLT), then exposed every one of them to an AI agent through an MCP server. The design goal throughout: an agent driving the app over MCP produces the exact same edit as a human — or a computer-use agent — clicking through the UI.

1.Mapping CapCut behavior into Shotcut

For each feature we started from CapCut's interaction model and found the closest idiomatic seam in Shotcut's Qt/C++/MLT stack, then matched the outcome (the resulting edit on export) rather than copying CapCut's exact panel layout.

UI/UX reference. We used the reference screenshots supplied in the take-home HTML (the CapCut panels for Transitions, Speed curve, Keyframes, Animation In/Out/Combo, Auto-captions, and Reduce-noise) as the source of truth for where each feature lives and what it should do. Each feature was implemented against its Shotcut dock so the UX mirrors the CapCut reference:

2.Architecture — a backend-driven, agent-first design

The central decision was to make the app backend-driven. All business logic for every feature lives in a single C++ control-plane service (EditorService) inside Shotcut. Both entry points — the human-facing Qt UI docks and the MCP server — are thin layers that call the exact same service methods. There is no logic that only exists in the UI, and none that only exists in the MCP path.

Human user AI agent (MCP / computer-use) │ │ ▼ ▼ Qt / QML docks mcp-shotcut (thin adapter, stdio JSON) (Transitions, Speed, │ JSON-RPC over WebSocket Keyframes, Animation, ▼ Text, Audio) ControlServer ws://127.0.0.1:3847 │ │ └───────────────┬───────────────────┘ ▼ C++ EditorService ← all business logic lives here │ ▼ MLT engine (timeremap, luma, affine, arnndn, …)

Why this matters for agents. Because the UI and MCP converge on EditorService, an edit made by an agent calling shotcut_apply_transition is byte-for-byte the same operation as a human choosing a preset in the Transitions dock and clicking Apply to Selected Join. That means:

  • A computer-use agent clicking the UI and a tool-calling agent using MCP land on the identical edit and the identical .mlt on save.
  • Every mutation has a matching read-back tool (get_state, get_transition, get_speed_curve, list_keyframes, get_audio_denoise, get_captions) so the agent can verify results — no blind writes.
  • The MCP server holds no state and no logic: it validates typed args and forwards to the service. New UI features become agent-operable almost for free.

3.The six features — status & scope

Honest scoping below. Four features are fully shipped; two are partial, with the reason for each cut.

#FeatureStatusWhat we shipped / why the gap
1Transitions Complete Drag-to-apply between two clips with adjustable duration. Dissolve + full wipe family (13 luma presets: bars, barn doors, diagonals, iris circle/box/double, clock). Apply / get / set-duration / remove, all via MCP and UI.
2Speed curve (speed ramp) Complete Non-linear time-remap via editable velocity curve. 6 presets (montage, hero, bullet, jump_cut, flash_in, flash_out) + constant speed + fully custom keyframe curve, with optional pitch compensation.
Known v1 limit: the curve remaps source frames within the existing segment; the clip's timeline length stays fixed rather than growing/shrinking.
3Keyframes Complete Add / edit / remove / list keyframes on position.x, position.y, scale, rotation, opacity, with per-keyframe interpolation. list_animatable_properties exposes the toggle-able set.
4Clip animations (In / Out / Combo) Complete 6 presets (fade, zoom_in, zoom_out, slide_left, slide_up, fade_zoom) applied as In, Out, or Combo with adjustable duration — built on the same keyframe engine as feature 3.
5Captions & text templates Partial Captions shipped: SRT import onto a real subtitle track, get/set cue text, and automatic burn-in to the preview. Auto-transcription is wired to Shotcut's Whisper hook but requires a local whisper-cli + model.
Why partial: getting live auto-captions working end-to-end (bundling/configuring the Whisper executable + model, wiring async job progress into the preview) is substantially more work; we shipped the deterministic import path instead. Animated text templates were out of the timebox.
6Audio: denoise & separation Partial Denoise shipped: RNN speech denoise with adjustable strength (native rnnoise Link when MLT provides it; FFmpeg avfilter.arnndn fallback on Homebrew builds), plus get/remove and save/reload persistence.
Why partial: stem separation (vocals vs. instruments) needs a dedicated ML model (e.g. Demucs/Spleeter) plus an inference runtime and track-splitting pipeline — far more work than denoise — so it was left out of scope.
Whisper / live captions: auto-transcription returns "Whisper is not configured; set whisper executable and model in Settings" on this build. Standing up live captions (shipping a Whisper binary + model, and streaming async transcription into the timeline) would have been a lot more work, so captions are demonstrated via the import + burn-in path.

4.Running the MCP server

Two processes: the Shotcut build (which hosts the control server) and the small Node MCP adapter. Public fork: github.com/hashir-ayaz/shotcut.

0 · Clone the fork

git clone https://github.com/hashir-ayaz/shotcut.git
cd shotcut

1 · Build & launch Shotcut

cmake -GNinja -DCMAKE_BUILD_TYPE=Release \
  -DCMAKE_PREFIX_PATH="/opt/homebrew/opt/qt;/opt/homebrew" \
  -DCMAKE_INSTALL_PREFIX="$HOME/development/shotcut/install" \
  -S . -B build
cmake --build build -j"$(sysctl -n hw.ncpu)"
cmake --install build
./run.sh            # required after every install

Confirm the log shows: ControlServer listening on ws://127.0.0.1:3847

2 · Start the MCP server

cd mcp-shotcut
npm install
SHOTCUT_WS_URL=ws://127.0.0.1:3847 npx tsx src/index.ts

Or let an MCP client launch it. Project config lives at .cursor/mcp.json:

{
  "mcpServers": {
    "shotcut": {
      "command": "npx",
      "args": ["tsx", "${workspaceFolder}/mcp-shotcut/src/index.ts"],
      "env": { "SHOTCUT_WS_URL": "ws://127.0.0.1:3847" }
    }
  }
}
